# 專案工作規則

## 核心行為原則
- 回覆使用者時請一律使用**繁體中文**。
- 嚴格禁止擅自刪除原始檔案；如需移除、覆蓋或大幅重構既有檔案，必須先取得使用者明確同意。
- 修改任何檔案前，必須先向使用者說明：準備修改哪個檔案、修改的核心內容與原因。
- 任務或對話完成後，最終回覆需附上清晰的處理摘要。
- 每次修改並更新本地 Markdown 檔案（如 `AGENTS.md`、`問題解決紀錄.md`、`未解決.md` 等）後，必須自動執行 Git commit 與 push，同步至 GitHub 遠端儲存庫。

## 任務執行與記憶管理流程
1. **事前檢查 (Pre-check)：**
   - 每次回應使用者問題或執行任務前，**必須**先完整讀取 `AGENTS.md`、`問題解決紀錄.md` 與 `未解決.md`，以掌握目前的環境脈絡與過去失敗的經驗。
2. **事後/階段性紀錄 (Logging)：**
   - **狀況 A：任務成功解決或完成分階任務**
     - 更新 `問題解決紀錄.md`。內容需以當下時間 (UTC+8) 做區隔，保留歷史軌跡。
     - 紀錄格式至少包含：`日期時間`、`問題背景與環境 (Context & Environment)`、`核心錯誤訊息/現象 (Symptoms & Error Logs)`、`最終解決方案 (The Solution)`、`根本原因分析 (Root Cause)`。
     - 若該任務先前存在於 `未解決.md`，請將其從中移除或標記為已結案。
   - **狀況 B：任務尚未解決或遇到技術瓶頸**
     - 更新 `未解決.md`。內容需以當下時間 (UTC+8) 做區隔，保留歷程。
     - 紀錄格式至少包含：`當前目標 (Target Goal)`、`目前的進度與現狀 (Current State)`、`已嘗試過的方法與結果 (Attempted Solutions & Failures)`、`推測的原因 (Hypotheses)`、`下一步方向 (Next Steps)`、`相關資源與檔案 (Resources & Links)`。
     - 不得將未完成的任務寫成已解決，必須誠實說明當前阻礙。
